工作区，暂存区，版本库

初始化:
	git init  文件夹名		# 表示管理该文件夹
	git init			# 表示当前路径全被管理
	
	
添加作者信息：
	git config --global user.email "aaa@qq.com"
	git config --global user.name "lqz"


	
查看状态：
	git status
	红色表示未被管理
	绿色表示提交到了暂存区


提交暂存区: 
	git add a.txt	把a提交到暂存区
	git add .		把当前路径下的所有提交暂存区


提交版本库：
	commit -m "注释，我新增了readme.txt"	# 把暂存区所有提交到版本库
	git checkout .	# 恢复到提交版本的位置

查看修改明细：
	git diff

撤销修改：
	这个要分三种情况
		1、修改了文件，但未提交到暂存区(文件还在工作区)
			git checkout readme.txt		# 丢弃工作区的修改
			或者 git resotre readme.txt
			
		2、文件已修改，且已经提交给了暂存区(add)
			使用一下任一条命令，让文件退出暂存区到工作区。然后在执行第一种操作。
			git restore --staged readme.txt
			git reset HEAD readme.txt
			
		3、文件已经提交(commit)
			直接使用版本回退即可
			
			
把暂存区的东西拉回到工作区：
	git rest .		# 原来是绿的变红了

在管理的根目录创建.gitignore


-语法：
	直接在.gitignore写文件夹名字，表示文件夹忽略，不被管理
	写文件名字，表示文件忽略不被管理
	如果向忽略某个文件夹下某个文件，直接写 文件夹/xxx.py,表示把文件夹后面的xxx.py忽略
	/dist 表示根路径下的dist文件夹，不被管理
	*.py 表示后缀名为py的文件都被忽略
	*.log* 表示忽略日志文件



以斜杠/开头表示目录；
以星号*通配多个字符；
以问号?通配单个字符
以方括号[]包含单个字符的匹配列表；
以叹号!表示不忽略(跟踪)匹配到的文件或目录；

此外，git 对于 .ignore 配置文件是按行从上到下进行规则匹配的，意味着如果前面的规则匹配的范围更大，则后面的规则将不会生效；

查看分支：
	git branch		# 查看所有分支分支是绿的，表示在当前分支上
	

创建分支：
	git branch dev	# 创建分支dev


创建并切换：
	git checkout -b dev


删除分支：
	# ps：删除分支首先不能在该分支上
	git branch -d dev	# 删除dev


切换分支：
	git checkout dev	# 切换到dev


合并分支：
	# 把dev分支合并到master分支：切换到master分支，执行合并dev分支的命令
	git merge 分支名	


提交本地分支到远程：
	git push origin dev


线上分支合并：
	在码云的代码库里面的pull request操作


出现冲突就两种情况：
	1、分支合并
		# 新建一个分支并切换
		git switch -c dev
		# 修改该分支的某个文件
		# 然后提交
		
		# 然后切换到master分支
		# 然后再将master分支上的文件修改提交
		
		# 然后两个分支合并就有冲突了
		# 这个时候需要手动把文件修改好
		# 然后添加暂存区，提交版本库，提交远端仓库(先pull，然后push)
	2、在同一个文件中改了同一行

# 1、公司内部大部分使用ssh连接
	- 公司给个git地址(项目地址)
	- 然后我们 git clone 地址到本地
	- 改代码，改完了--》提交(禁止)
	- ssh配置，以后都不用输密码了
	
	
# 2、配置如何做
	- 对称加密 (加密和解密用同一套密码)
	- 非堆成加密，公钥加密，私钥解密
	- 生成一套密钥(用命令)
		ssh-keygen -t rsa -C "lqz@qq.com"	生成到用户家目录的.ssh文件夹下(一个公钥，一个私钥)
	- 把公钥复制出来，在码云上配置
